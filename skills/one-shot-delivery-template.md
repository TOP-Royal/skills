# 一次成型交付编排模板（one-shot-delivery）

> 用途：你只提供一次“原始需求”，AI 必须按顺序输出：PRD → 设计约束 → API/DB/安全/观测（按需）→ 测试策略 → DoD。

## 文档门禁（强制）

- 未完成 `1.1 PRD`：禁止进入实现方案与代码阶段
- 未完成 `1.2 编码前约束检查（Preflight）`：禁止开始写代码
- 未完成 `1.4 测试策略`：禁止提交 PR
- 未完成 `1.5 DoD`：禁止合并/发布

---

## 0. 原始需求输入区（你只需要填这里）

- 一句话需求：
- 目标用户：
- 关键流程（最多 5 步）：
- 必须具备的功能点（3-8 条）：
- 不做什么（非目标）：
- 技术栈/项目类型（可选）：
  - 前端：
  - 后端：
  - 数据库/缓存/消息队列：
- 约束（可选）：时间/性能/合规/预算

---

## 1. AI 输出要求（强制执行）

AI 必须按以下顺序输出，并带清晰标题：

### 1.0 本次交付物清单（先列清楚再展开）

- PRD（需求说明书）
- Preflight（实现前约束与风险清单）
- 护栏文档（按需：API 设计 / DB 设计 / 安全编码 / 可观测性）
- 测试策略与用例表
- DoD（提交前验收清单）

### 1.1 产品思维补全（需求不明确时用 `product-manager`）
- 触发：用户只说"要做XX功能"，但没有用户画像/场景/竞品/价值评估/路线图
- 输出：用户画像/场景故事/竞品分析/价值评估/产品路线图/数据指标/风险
- 禁止：需求不明确时直接进入 PRD（先补产品思维）

### 1.2 设计规范补全（涉及UI/UX/前端还原时用 `designer`）
- 触发：有页面/组件/交互/动画/响应式/无障碍需求，但无设计系统/状态定义/交互流程
- 输出：设计系统Tokens/组件状态表/交互流程图/响应式策略/动画规范/无障碍检查
- 禁止：无设计规范直接写前端代码（先补设计系统 + 状态 + 交互）

### 1.3 PRD（使用 `requirements-doc`）
- 输出：背景/目标/范围/流程/功能需求/数据与规则/NFR/验收标准/测试/发布回滚/Open Questions

### 1.4 编码前约束检查（使用 `coding-preflight`）
- 输出：输入输出、失败场景、幂等/并发/事务、超时重试、观测、测试计划

### 1.5 护栏模块（按触发条件选择输出）

- 触发：有对外/前后端接口 → 输出 `api-design`
- 触发：有表结构/查询/迁移 → 输出 `db-design`
- 触发：技术选型/引入开源依赖/找替代品 → 输出 `tech-research`（含 OpenSSF Scorecard Gate 门禁）
- 触发：涉及外部输入/权限/敏感数据/上传 → 输出 `secure-coding`
- 触发：关键链路/线上定位困难 → 输出 `observability`
- 触发：生产事故/告警升级/故障演练 → 输出 `sre-devops`（Incident 角色分工/沟通渠道/状态通告模板/联系人清单/判定标准）

### 1.6 测试策略（使用 `testing-strategy`）
- 输出：分层测试计划 + 用例（正常/异常/边界）+ 质量门槛

### 1.7 提交前 DoD（使用 `definition-of-done`）
- 输出：可勾选清单，明确当前变更是否满足
- 额外门禁：安全敏感变更必须完成 SDL 分阶段门禁自检（Design/Code/Build&Deploy/Run）

---

## 2. 输出格式要求

- 使用 Markdown
- 关键表格：错误码表、字段/Schema 表、测试用例表
- 所有“未决问题”集中列在最后
