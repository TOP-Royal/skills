# 数据分析与 BI 能力模板（data-analyst）

> 目标：从原始数据到可决策的指标与可视化，覆盖数据接入、清洗、建模、指标定义、报表与可视化、数据治理。

---

## 固定输出物（AI 必须产出）

- 数据源与接入方案清单（类型/频率/格式/增量策略/延迟要求）
- 数据模型与维度建模（事实表/维度表/星型/雪花/宽表）
- 核心指标体系（北极星指标 + 过程指标 + 维度拆解 + 计算公式）
- ETL/ELT  pipeline 设计（抽取/转换/加载/调度/依赖/容错）
- BI 报表与可视化方案（受众/频率/交互/权限/移动端适配）
- 数据质量规则（完整性/准确性/一致性/及时性/唯一性检查）
- 数据安全与合规（分级/脱敏/权限/审计/保留策略）

## 必须/禁止规则

- 必须：所有指标有明确的业务定义、计算公式、数据来源、更新频率、负责人
- 必须：维度建模遵循一致的业务口径，同名指标在不同报表中定义一致
- 必须：数据 pipeline 有依赖管理和重跑机制，失败可自动告警与补偿
- 必须：敏感数据（PII/交易/用户行为）分级处理，最小权限原则
- 禁止：报表指标无业务定义就直接展示（避免“这个数字是什么意思”）
- 禁止：生产数据库直连做 BI 查询（必须走只读副本/数仓/数据湖）
- 禁止：忽略数据质量（脏数据导致决策错误比没数据更危险）
- 禁止：把原始日志/埋点直接当指标（必须经过清洗和口径统一）

## 1. 数据源与接入

### 1.1 数据源分类

| 类型 | 示例 | 接入方式 | 频率 | 延迟要求 |
|------|------|---------|------|---------|
| 业务数据库 | MySQL/PostgreSQL/MongoDB | CDC / 定时抽取 | 实时/小时/天 | 分钟级/小时级 |
| 埋点/日志 | Web/App/服务端日志 | Kafka/Fluentd/Logstash | 实时 | 秒级 |
| 第三方数据 | 广告平台/支付/CRM | API / Webhook / SFTP | 小时/天 | 小时级/天级 |
| 文件/Excel | 运营/财务手工数据 | 上传/S3/邮件附件 | 天/周 | 天级 |

### 1.2 增量策略

- CDC（Change Data Capture）：Debezium / Flink CDC / Canal
- 时间戳增量：`WHERE updated_at > last_sync_time`
- 自增 ID 增量：`WHERE id > last_max_id`
- 全量快照：小表/参考数据/每日兜底
- 禁止：无唯一键或更新时间戳的表做增量同步

## 2. 数据建模

### 2.1 数仓分层

| 层级 | 命名 | 内容 | 使用者 |
|------|------|------|--------|
| ODS | 操作数据存储 | 原始数据，与源系统一致 | 数据工程师 |
| DWD | 明细数据层 | 清洗后的事实数据 | 数据工程师/分析师 |
| DWS | 汇总数据层 | 轻度聚合（按日/周/主题） | 分析师/BI 开发 |
| ADS | 应用数据层 | 面向报表/应用的宽表 | BI/产品/运营 |
| DIM | 维度层 | 用户/商品/地域/时间维度 | 全链路 |

### 2.2 建模方法

- 维度建模（Kimball）：星型/雪花，适合 BI 查询
- Data Vault：hub/link/satellite，适合复杂变更追踪
- One Big Table（宽表）：适合大数据量快速查询（ClickHouse/BigQuery）
- 选择依据：查询模式/数据量/变更频率/团队熟悉度

## 3. 核心指标体系

### 3.1 指标定义模板

```
指标名：
业务定义：
计算公式：
数据来源（表/字段）：
维度拆解：
更新频率：
负责人：
```

### 3.2 指标类型

- 原子指标：不可拆分的最小度量（如订单金额、登录次数）
- 派生指标：原子指标 + 修饰词（如"昨日移动端新客订单金额"）
- 复合指标：多个指标计算（如客单价 = GMV / 订单数）

### 3.3 维度设计

- 一致性维度：时间/用户/商品/地域/渠道/设备/版本
- 维度退化：高频查询维度直接冗余到事实表
- 缓慢变化维：Type 1（覆盖）/ Type 2（历史追踪）/ Type 3（部分历史）

## 4. ETL/ELT Pipeline

### 4.1 工具选型

- 调度：Airflow / Dagster / Prefect / dbt + cron
- 计算：Spark / Flink / dbt / SQL（BigQuery/Snowflake/ClickHouse）
- 存储：数据湖（S3/HDFS/Delta Lake/Iceberg）+ 数据仓库（Snowflake/BigQuery/ClickHouse/StarRocks）

### 4.2 Pipeline 设计原则

- 幂等：同一任务重跑不产生重复数据（INSERT OVERWRITE / MERGE / UPSERT）
- 依赖：DAG 显式声明上游依赖，失败级联阻断下游
- 容错：
  - 重试：失败自动重试 3 次（指数退避）
  - 告警：Slack/钉钉/企业微信通知
  - 补偿：失败任务有手动重跑入口和校验脚本
- 性能：
  - 大表分区（按日期/地区）
  - 小表广播 join
  - 避免全表扫描

## 5. BI 报表与可视化

### 5.1 报表分类

| 类型 | 受众 | 频率 | 交互 | 工具 |
|------|------|------|------|------|
| 管理层驾驶舱 | CEO/高管 | 实时/日 | 少（看数） | 大屏/固定报表 |
| 运营日报 | 运营/产品 | 日 | 中（筛选/下钻） | BI 工具 |
| 自助分析 | 分析师/PM | 按需 | 高（拖拽/自定义） | BI 工具/SQL |
| 移动端 | 管理层/外勤 | 实时/日 | 少 | 小程序/App 嵌入 |

### 5.2 工具选型

- BI 平台：Tableau / Power BI / Looker / Metabase / Superset / FineBI
- 可视化库：ECharts / AntV / D3.js / Plotly
- 嵌入式：iframe / SDK / API（数据脱敏后）

### 5.3 设计原则

- 一页一主题，避免信息过载
- 关键指标置顶，支持同比/环比/目标对比
- 下钻路径清晰（总览 → 明细 → 原始数据）
- 权限：行级/列级/报表级（与组织架构/业务线绑定）

## 6. 数据质量

### 6.1 质量维度

| 维度 | 检查项 | 示例 |
|------|--------|------|
| 完整性 | 非空率 / 记录数波动 | 订单表每日记录数不应突降 50% |
| 准确性 | 数值范围 / 枚举值 / 关联一致性 | 订单金额 ≥ 0，状态在枚举范围内 |
| 一致性 | 跨系统口径一致 |  BI  GMV = 财务 GMV（误差 < 1%） |
| 及时性 | 数据到达延迟 | ODS 层应在 T+1 08:00 前就绪 |
| 唯一性 | 主键/业务键唯一 | 订单 ID 全局唯一 |

### 6.2 监控与告警

- Great Expectations / dbt tests / Monte Carlo / Soda
- 规则：每个核心表至少 3 条质量规则（完整性/准确性/及时性）
- 告警：质量异常 → 阻塞下游 + 通知负责人

---

## 7. 数据安全与合规

- 数据分级：
  - 公开 / 内部 / 敏感 / 机密
  - PII（个人身份信息）：姓名/电话/身份证/地址
  - 金融数据：卡号/交易金额/账户余额
- 脱敏：
  - 静态脱敏：开发/测试环境使用脱敏数据
  - 动态脱敏：BI 查询时实时脱敏（如电话 138****1234）
- 权限：
  - 最小权限原则
  - 审批流程（敏感数据访问需审批）
  - 审计日志：谁/何时/访问了什么数据
- 保留策略：
  - 日志保留 90-180 天
  - 原始埋点保留 1-2 年
  - PII 按法规要求删除（用户注销后 30 天内）
