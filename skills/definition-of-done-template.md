# 完成验收模板（Definition of Done / DoD）

> 目标：写完代码后用清单把错误挡在提交前，尽量做到“一次成型”。每次提交 PR 前复制本模板自检。

## 门禁（强制）

- 未关联 PRD/需求说明：不允许合并
- 未覆盖验收标准（AC）的测试：不允许合并
- 无回滚方案/无关键观测：不允许发布

---

## 1. 功能正确性

- [ ] 核心业务路径已覆盖（主流程 + 关键分支）
- [ ] 失败场景均有明确行为（错误码/提示/重试/降级）
- [ ] 边界值已验证（空、极大/极小、非法字符、超长文本、特殊时间）
- [ ] 幂等性验证通过（重复请求/重复事件不会产生副作用）
- [ ] 并发场景验证通过（不会产生脏写/丢失更新）

---

## 2. API / 契约（如有）

- [ ] 请求/响应 schema 已更新（OpenAPI/Swagger/Protobuf）
- [ ] 向后兼容策略明确（版本、默认值、字段弃用）
- [ ] 错误响应格式统一（code/message/detail/request_id）
- [ ] 超时、限流、分页、排序、过滤行为明确且一致

---

## 3. 数据库与迁移（如有）

- [ ] 迁移脚本提供 up/down
- [ ] 回滚可执行且验证过（至少演练一次）
- [ ] 索引设计合理（避免全表扫、避免无用索引）
- [ ] 事务边界正确（不会长事务、不会锁表）
- [ ] 数据一致性策略明确（强一致/最终一致）

---

## 4. 可观测性

- [ ] 关键路径有结构化日志（包含 request_id/trace_id + 关键业务字段）
- [ ] 日志无敏感信息泄露（PII/Token/密码脱敏）
- [ ] 指标已埋点（成功/失败计数 + 耗时直方图）
- [ ] 关键链路可追踪（trace/span 贯穿）
- [ ] 错误信息可定位（错误码能直接定位模块/原因）

---

## 5. 测试

- [ ] 单元测试已补齐（覆盖关键分支）
- [ ] 集成测试覆盖外部依赖（DB/Redis/MQ/第三方 API）
- [ ] E2E 测试覆盖核心用户路径（如有 UI）
- [ ] 测试在 CI 里可稳定运行（无随机失败）

---

## 6. 代码质量

- [ ] Lint/Format/Typecheck 全部通过
- [ ] 无明显代码坏味道（重复逻辑、巨函数、魔法常量）
- [ ] 命名清晰（变量/函数/接口语义明确）
- [ ] 错误处理完整（不吞异常、不返回模糊错误）
- [ ] 资源释放正确（文件句柄、连接、锁、goroutine/task）

---

## 7. 性能与可靠性

- [ ] 设置了合理的超时（外部请求、数据库操作、队列消费）
- [ ] 重试策略不会放大故障（退避、熔断、上限）
- [ ] 缓存策略合理（命中率、失效策略、雪崩/穿透防护）
- [ ] 性能基线可接受（至少本地压测或基准测试）

---

## 8. 安全

- [ ] 输入校验完备（防注入、防越权、防路径遍历）
- [ ] 授权检查在服务端完成（不信任前端）
- [ ] 依赖漏洞扫描无高危（或有明确豁免理由）
- [ ] Secret 不落库、不进日志、不提交到仓库

### 8.1 安全开发门禁（参考 Microsoft SDL，按阶段自检）

- **Design（设计阶段）**
  - [ ] 已完成安全设计评审/威胁建模（关键资产、攻击面、信任边界、滥用场景）
  - [ ] 明确最小权限与权限边界（用户/服务账号/第三方 Token）
  - [ ] 加密与密钥策略明确（加密算法、密钥轮换、KMS/Keychain/Secrets Manager）

- **Code（编码阶段）**
  - [ ] 使用成熟安全特性/框架（避免自研加密/自研鉴权）
  - [ ] 安全测试已覆盖（SAST/依赖扫描/常见漏洞检查）
  - [ ] 高风险变更已做安全走查（上传、支付、权限、外部输入、模板渲染）

- **Build & Deploy（构建与发布阶段）**
  - [ ] 供应链安全：依赖来源可信、锁文件/固定版本、构建产物可追溯
  - [ ] CI Token 权限最小化（只读优先），敏感环境变量不暴露
  - [ ] 构建环境安全：不在不可信 runner 上构建敏感产物（如签名包）

- **Run（运行阶段）**
  - [ ] 运行环境基线安全（最小镜像、关闭多余端口、最小权限运行）
  - [ ] 安全监控与响应：关键安全事件有告警（登录失败暴增、权限异常、WAF/IDS 告警）
  - [ ] 具备应急处置路径（密钥泄露轮换、封禁账号、降级开关、回滚）

---

## 9. 发布与回滚

- [ ] Feature Flag / 灰度策略准备好（如需要）
- [ ] 回滚路径明确（配置回滚/版本回滚/数据回滚）
- [ ] 监控面板与告警规则就绪（错误率、延迟、吞吐）
- [ ] 变更说明写清楚（影响范围、风险点、验证方式）

---

## 10. PR 自检（提交前最后一步）

- [ ] PR 描述包含：背景/方案/测试/风险/回滚
- [ ] 变更尽量小且聚焦（不夹带无关重构）
- [ ] 关键代码有足够上下文（不要只留“神秘函数”）
- [ ] 新增依赖有理由，版本明确
- [ ] 已本地完整跑一遍关键流程
