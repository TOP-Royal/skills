# 测试策略模板（testing-strategy）

> 目标：在写代码前把测试策略定下来，让 bug 在提交前暴露。

## 固定输出物（AI 必须产出）

- 测试分层覆盖矩阵（变更点 → Unit/Integration/E2E/Contract）
- 测试用例表（至少：正常 3、异常 5、边界 5）
- 质量门槛（覆盖率目标、CI 必跑项、禁止项）

## 必须/禁止规则

- 必须：关键路径至少有集成测试或 E2E（择一，不可都没有）
- 必须：每个线上 bug 都补回归用例
- 禁止：只靠手工点点点就上线

## 2. 测试分层

- 单元测试（Unit）：
  - 覆盖：纯函数/业务规则/边界条件
  - Mock：外部依赖（DB/HTTP/MQ）
- 集成测试（Integration）：
  - 覆盖：DB/缓存/队列/第三方接口
  - 建议：Testcontainers / docker-compose
- 端到端测试（E2E）：
  - 覆盖：核心用户路径（登录→下单→支付）
- 契约测试（Contract）：
  - 覆盖：前后端/服务间协议（OpenAPI/Protobuf）

---

## 3. 用例设计

至少写：
- 正常流程 3 条
- 异常流程 5 条
- 边界值 5 条

---

## 4. 质量门槛

- 覆盖率目标：行覆盖率/分支覆盖率（给出数值）
- CI 必跑：lint + typecheck + unit + integration
- 禁止：只靠手工测试、没有回归用例

---

## 5. 验收清单（DoD）

- [ ] 关键路径有 E2E 或等价集成测试
- [ ] 每个 bug 都有对应回归用例
- [ ] CI 稳定通过（无偶发失败）
