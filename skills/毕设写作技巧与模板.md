# 毕业论文写作技巧与模板指南

## 一、论文整体结构模板

计算机类毕设通常按以下章节组织：

```
第1章 绪论
  1.1 研究背景与意义
  1.2 国内外研究现状
  1.3 研究内容与目标
  1.4 论文组织结构

第2章 相关技术与理论基础
  2.1 技术概述
  2.2 关键技术介绍
  2.3 本章小结

第3章 系统需求分析与设计
  3.1 可行性分析
  3.2 需求分析（功能/非功能）
  3.3 功能模块设计
  3.4 数据库设计
  3.5 本章小结

第4章 系统实现
  4.1 开发环境
  4.2 主要程序模块实现（按功能模块分节）
  4.3 前端实现
  4.4 系统运行与效果展示
  4.5 本章小结

第5章 系统测试与结果分析
  5.1 测试环境
  5.2 测试方法与用例设计
  5.3 测试运行和测试记录
  5.4 测试结果分析
  5.5 本章小结

第6章 总结与展望
  6.1 工作总结
  6.2 未来展望

致谢
参考文献
附录（可选）
```

## 字数分配参考（本科毕设 1.2~1.5 万字）

| 章节 | 建议字数 | 说明 |
|------|---------|------|
| 摘要（中文） | 300~500 字 | 背景+方法+结果+结论 |
| 摘要（英文） | 250~400 词 | 与中文对应 |
| 第1章 绪论 | 2000~2500 字 | 背景现状可稍多 |
| 第2章 相关技术 | 2500~3000 字 | 每个技术 600~800 字 |
| 第3章 需求分析与设计 | 2500~3000 字 | 图表多，文字可精简 |
| 第4章 系统实现 | 3000~4000 字 | 核心章节，代码+说明 |
| 第5章 系统测试 | 2000~2500 字 | 用例表格占篇幅 |
| 第6章 总结与展望 | 800~1000 字 | 不要太短 |
| 参考文献 | 15~25 篇 | 技术类+行业类结合 |

## 摘要写作模板

摘要放在目录之前，中英文各一份。结构固定为 **"背景→问题→方法→结果→结论"**，不要出现"本文""笔者"等第一人称，不要引用参考文献。

**中文摘要模板**：
> 随着我国老龄化程度不断加深，中小型养老机构在日常管理中面临信息分散、流程不规范与统计困难等问题。针对上述现状，本文设计并实现了一套基于 Spring Boot 与 Vue.js 的敬老院管理系统。系统采用前后端分离的 B/S 架构，后端使用 Spring Boot 结合 MyBatis-Plus 实现数据持久化与业务逻辑处理，前端使用 Vue 3 与 Element Plus 构建管理界面，并通过 ECharts 实现数据可视化。系统实现了长者档案管理、房间与床位管理、入住退住管理、来访登记、餐饮服务与统计报表等核心功能。经功能测试，各模块业务流程正确、数据一致，能够满足中小型敬老院日常信息化管理需求。

**英文摘要模板**：
> With the deepening of population aging, small and medium-sized nursing institutions face problems such as scattered information, non-standard processes, and difficult statistics in daily management. To address these issues, this thesis designs and implements a nursing home management system based on Spring Boot and Vue.js. The system adopts a front-end and back-end separated B/S architecture...

**关键词**：3~5 个，如：敬老院管理；Spring Boot；Vue.js；前后端分离；信息化系统

## 二、各章写作技巧

### 第1章 绪论

**核心作用**：让读者快速理解"为什么要做这个系统"。

**本章引言写法**（放在"1.1"之前）：
> 本章主要阐述我国养老服务业发展现状与养老机构信息化建设的现实需求，分析传统养老管理方式存在的效率低、信息分散与管理难等问题，明确本系统研发的现实意义；综述国内外养老管理系统与相关技术的研究进展，指出现有系统的不足；说明本文的研究内容、设计思路与预期成果，并介绍论文整体组织结构，为后续研究奠定整体框架。

- **1.1 研究背景与意义**：从宏观到微观。先写行业现状（如老龄化加剧→养老机构增多→信息化需求），再点出传统管理痛点（手工记录效率低、信息孤岛、统计困难），最后说明本系统的实际意义。
- **1.2 国内外研究现状**：列举2-3个同类产品/系统，分析其优点和不足，引出本系统要解决的问题。
- **1.3 研究内容与目标**：用条目式列出主要功能模块（如长者档案、入住管理、餐饮管理等），说明预期成果。
- **1.4 论文组织结构**：逐章简述每章内容，固定句式可参考：
  > "第1章为绪论，介绍研究背景、现状与论文结构；第2章介绍系统开发所用的关键技术；第3章进行需求分析与系统设计；第4章详细阐述系统实现过程；第5章对系统进行测试并分析结果；第6章总结全文并展望未来。"

**技巧**：绪论不要涉及具体代码和界面，保持"概括性"和"学术性"。

**本章小结写法**：
> 本章从研究背景出发，分析了传统敬老院管理方式的不足与信息化建设的必要性，并对国内外相关系统与技术进行了综述。在此基础上明确了本文的研究目标与主要内容，最后给出了论文的整体组织结构安排。

---

### 第2章 相关技术

**核心作用**：证明你掌握了开发本系统所需的技术栈。

- 每个技术单独成节（如 2.1 Spring Boot、2.2 Vue.js、2.3 MyBatis-Plus）
- 每节结构：技术简介 → 核心特性 → 在本系统中的作用
- **不要只抄百度百科**，要联系项目说明选择该技术的原因

**参考模板**：
> "Spring Boot 是 Pivotal 团队推出的基于 Spring 框架的快速开发框架。它采用"约定优于配置"的设计理念，简化了 Spring 应用的初始搭建和开发过程。本系统后端采用 Spring Boot 构建 RESTful API 服务，利用其自动配置特性快速集成 MySQL 数据源和事务管理。"

**本章小结写法**：
> 本章介绍了系统开发过程中涉及的主要技术栈。首先概述了 Spring Boot 后端框架的核心特性与自动配置机制；其次介绍了 Vue.js 前端框架的组件化开发模式；最后说明了 MyBatis-Plus、JWT 与 MySQL 在数据访问、身份认证与持久化存储中的作用。这些技术为后续系统的设计与实现提供了基础支撑。

---

### 第3章 需求分析与设计

**核心作用**：证明系统不是拍脑袋做的，而是经过合理规划的。

#### 3.1 可行性分析
- 技术可行性（技术栈成熟，有学习资源）
- 经济可行性（开源技术，无授权费用）
- 操作可行性（界面友好，培训成本低）

#### 3.2 需求分析
- **功能需求**：用用例图（Use Case）展示系统参与者（管理员、工作人员）和功能模块关系
- **非功能需求**：性能（响应时间<2s）、安全性（密码加密、权限控制）、可用性（支持主流浏览器）

#### 3.3 功能模块设计
- 按模块逐一描述，每个模块说明功能点
- 与第4章实现形成对应关系

#### 3.4 数据库设计
- 先画 ER 图，再给出每个表的详细字段说明
- 表格式：`表 X-X 表名（英文表名）`
- 每个表配一段文字说明其用途和关联关系

**技巧**：第3章多用图表（用例图、ER图、流程图），减少大段文字。

**数据库表描述格式**（每表一段）：
```
（X）表中文名
表中文名用途说明文字。该表的结构设计如表 X-X 表中文名（英文表名）所示。

**表 X-X 表中文名（英文表名）**
| 字段名 | 中文含义 | 数据类型 | 是否主键 | 是否为空 |
|--------|----------|----------|----------|----------|
| id | ID | bigint | 是 | 否 |
```
**注意**：表标题和描述文字之间**不要空行**。

**本章小结写法**：
> 本章在需求分析的基础上，完成了系统的总体功能模块划分与数据库结构设计。首先进行了技术、经济与操作可行性分析；其次通过用例图描述了系统的功能需求；然后按模块说明了各功能点的设计思路；最后给出了系统数据库的 ER 图与各表详细结构。

---

### 第4章 系统实现

**核心作用**：展示系统的核心功能和关键代码。

#### 4.1 开发环境
简要列出软硬件环境：操作系统、IDE、JDK版本、Node.js版本、数据库版本等。

#### 4.2 主要程序模块实现（重点）
**每个模块的标准写法**：

```
4.2.X 模块名模块实现
（1）业务流程：用文字描述操作步骤（谁做了什么→系统做了什么）
（2）流程图：如图 X-X 模块业务流程图所示。（对应你的UML流程图）
（3）核心代码/核心逻辑：给出关键代码段或伪代码，配合注释说明
```

**代码展示技巧**：
- 不要贴完整文件，只贴核心逻辑（2-3个关键方法）
- 代码前要说明这段代码解决了什么问题
- 伪代码也可以，重在展示思路
- 关键变量和判断条件加注释

#### 4.3 前端实现
- 简述前端框架和组件库选择
- 描述主要页面布局（如左侧菜单+右侧内容区）
- 提及2-3个典型页面的交互流程（如登录页→首页→列表页→弹窗表单）

#### 4.4 系统运行与效果展示
- 放关键界面截图（登录页、列表页、图表页等）
- 每张图配简要说明文字
- 可以补充接口调用示例（如Postman截图）

**本章小结写法**：
> 本章介绍了系统的实现环境与各主要程序模块的实现过程。首先说明了开发所需的软硬件环境；其次以入住管理、长者档案、房间管理等模块为例，给出了业务流程描述、核心伪代码与运行效果截图；最后展示了前端页面布局与系统整体运行效果。经测试，各模块功能符合设计预期。

---

### 第5章 系统测试

**核心作用**：证明系统功能正确、能正常使用。

#### 5.1 测试环境
与4.1类似，但侧重测试工具（Postman/Apifox、浏览器版本）。

#### 5.2 测试程序与测试数据设计
- 说明测试方法（黑盒测试为主）
- 说明测试数据准备（预置多少条长者、房间记录等）
- 提及异常场景设计（如重复入住、删除在住长者等）

#### 5.3 测试运行和测试记录（重点）
**每个模块的标准写法**：

```
5.3.X 模块名测试
    [一段描述性文字，说明这个模块测试了什么、覆盖哪些场景]

表 5-X 模块测试用例表
用例编号 | 用例标题 | 功能模块 | 优先级 | 前置条件 | 测试步骤 | 预期结果 | 实际结果
```

**测试用例设计技巧**：
- 每个模块至少2-3条用例，覆盖正常流程和异常流程
- 优先级：P0（核心流程必测）、P1（异常场景）
- 前置条件要写清楚（如"已登录系统"、"某房间已满员"）
- 实际结果没测试前可以写"待测试"，或写"与预期一致"

#### 5.4 测试结果分析
- 总体评价：功能是否正常、异常处理是否完善
- 分模块简述：哪些通过、有无遗留问题
- 性能简评：响应速度、并发能力（本科毕设不需要压测数据）

**本章小结写法**：
> 本章对系统进行了功能测试与结果分析。首先说明了测试环境与测试数据准备情况；其次按模块列出了测试用例并记录了执行结果；最后对测试结果进行了总结，表明系统核心功能正确、异常处理完善，能够满足设计目标。

---

### 第6章 总结与展望

**模板**：
> "本文设计并实现了一套基于 XXX 技术的 XXX 管理系统。系统采用前后端分离架构，实现了 XXX、XXX 等核心功能，满足了 XXX 的业务需求。通过本次毕业设计，作者对 XXX 技术有了更深入的理解，也积累了实际项目开发经验。系统的不足之处包括 XXX，未来可在以下方面优化：XXX。"

## 三、格式与排版技巧

### 1. 图表规范
- **图**：图下方标注"图 X-X 图名"，如"图 3-1 系统功能模块图"
- **表**：表上方标注"表 X-X 表名"，如"表 3-1 系统用户表"
- 图表都要在正文中引用："如图 3-1 所示"

### 2. 图表素材管理（建议单独存放）

**流程图/UML 图**：
- 建议用 **PlantUML** 写代码，单独存一个 `.puml` 文件
- 好处：后续改需求或调整流程时，改代码重新生成即可，不用重新画图
- 在线生成地址：https://www.plantuml.com/plantuml
- VS Code 可装 PlantUML 插件实时预览
- 每个模块一个 `@startuml ... @enduml` 块，注释标注对应图号
 
**数据库 ER 图、架构图**：
- 可用 ProcessOn、draw.io 或 Visio 绘制
- 导出为 PNG/SVG，同样单独存放并命名对应图号

### 3. 编号层级
```
第1章 → 1.1 → 1.1.1（最多三级）
```

### 4. 代码与伪代码
- 用等宽字体或代码块展示
- 重要逻辑加行内注释
- 代码过长时只截取关键部分，注明"...（省略其他代码）"

### 5. 参考文献格式
- 按学校要求（通常是GB/T 7714格式）
- 顺序编码制：[1]、[2] 按正文中出现顺序排列
- 常见类型：
  - 书籍：[序号] 作者.书名[M].出版地:出版社,年份.
  - 期刊：[序号] 作者.文章名[J].期刊名,年份,卷(期):页码.
  - 学位论文：[序号] 作者.题名[D].保存地:保存单位,年份.

## 四、常见避坑指南

| 问题 | 解决方案 |
|------|---------|
| 论文字数不够 | 第2章技术介绍写详细点；第3章多画图表配文字说明；第5章测试用例多写几条 |
| 前后章节不对应 | 第3章的功能模块名称要和第4章、第5章完全一致（如都是"入住管理"，不要第3章叫"入住管理"，第4章叫"入住退住模块"） |
| 代码太多占篇幅 | 用伪代码代替；只贴关键方法；删除与项目无关的通用代码（如工具类） |
| 截图模糊 | 用浏览器开发者工具或Snipaste截取，确保分辨率足够 |
| 查重率高 | 技术介绍用自己的话重新组织，不要大段复制百度百科；代码用伪代码或精简后展示；同一段话不要连续超过13个字与网络资源相同 |
| 参考文献太少 | 至少15-20篇，涵盖技术类（Spring Boot、Vue.js）和行业类（养老信息化） |
| 每章开头缺引言 | 可以在每章第一节（如1.1、3.1、4.1、5.1）之前加一段"本章引言"，总领全章 |
| 每章结尾缺小结 | 每章最后一节固定写"本章小结"，总结本章做了什么、为下一章铺垫什么 |

## 五、快速检查清单

提交前逐项确认：

- [ ] 目录页码与实际页码一致
- [ ] 图表编号连续，无跳号
- [ ] 正文中所有"如图 X-X 所示"都能对应上实际图片
- [ ] 第3、4、5章模块名称一一对应
- [ ] 参考文献格式统一，正文中都有引用标注
- [ ] 代码能运行（答辩时可能要现场演示）
- [ ] 摘要（中英文）包含研究背景、方法、结果、结论
- [ ] 致谢不要复制网上的，用自己的话写
