# 消息队列模板（message-queue）

> 目标：消息可靠性、顺序性、幂等消费、死信处理一次设计正确。

---

## 固定输出物（AI 必须产出）

- Topic/Queue 清单（命名规范、分区数、副本数、 retention）
- 生产者可靠性方案（同步发送 / 异步回调 / 事务消息 / 本地事务表）
- 消费者幂等设计（去重 key / 数据库唯一约束 / 幂等表 / 业务状态机）
- 消费顺序策略（单分区 / 分区 key / 局部有序）
- 死信队列与重试策略（重试次数、退避间隔、人工介入入口）

## 必须/禁止规则

- 必须：消息消费幂等（网络抖动/重平衡导致重复消费）
- 必须：生产者发送失败有日志+告警+补偿（禁止静默丢弃）
- 必须：消费异常进死信队列，禁止无限重试阻塞队列
- 禁止：跨服务的强一致性依赖消息（消息最终一致，强一致走同步接口）
- 禁止：消费逻辑里直接调外部 HTTP 且不超时/不重试
- 禁止：消费端大事务（尽量小事务，失败可重放）

---

## 1. 选型与 Topic 设计

- 选型对比：
  - Kafka：高吞吐、日志型、保留时间长、多分区高并发
  - RabbitMQ：路由灵活、延迟队列、优先级、事务消息
  - RocketMQ：金融级可靠、事务消息、顺序消息、延迟消息
  - Pulsar：云原生、多租户、分层存储、跨地域复制
- Topic 命名：`业务域.事件类型`（如 `order.pay.success`）
- 分区策略：
  - 默认：轮询/随机
  - 顺序：按 key hash（如 user_id / order_id）
- Retention：按时间/按大小（合规要求需长期保留的单独 topic）

---

## 2. 生产者可靠性

- 发送方式：
  - 同步：重要消息（支付、订单），等 ack
  - 异步：日志、监控，callback 处理异常
  - 事务：本地 DB + 消息事务（RocketMQ / Kafka EOS）
- 失败处理：
  - 重试：指数退避，最大重试次数
  - 降级：本地日志 + 定时补偿任务
  - 告警：连续失败触发告警
- 禁止：fire-and-forget 且无异常处理

---

## 3. 消费者设计

- 消费模式：
  - 集群消费：一条消息只被一个实例消费（默认）
  - 广播消费：所有实例都收到（配置变更通知）
- 并发控制：
  - 线程池大小
  - 消费速率限制（防 DB 被打爆）
- 异常处理：
  - 业务异常：进死信队列（DLQ）
  - 系统异常：重试（间隔递增）
  -  poison pill：识别并隔离，防止阻塞

---

## 4. 幂等消费（核心）

- 去重 key：msg_id / biz_id（业务唯一键）
- 存储方式：
  - 数据库唯一约束（幂等表：msg_id + status）
  - Redis SETNX（TTL > 消费周期）
  - 业务状态机（状态流转幂等，如 pending → paid）
- 并发控制：
  - 分布式锁（Redis/ZK）：同 key 串行处理
  - 乐观锁：version / update_time

---

## 5. 顺序消息（如需要）

- 全局顺序：单分区（吞吐量受限）
- 局部顺序：按 key 分区（如同一订单的消息在同一分区）
- 实现：
  - 生产者：相同 key 发送到同一分区
  - 消费者：单线程消费（或内存队列按 key 串行）
- 注意：分区数变更会破坏顺序，需谨慎

---

## 6. 死信队列与监控

- 死信队列（DLQ）：
  - 触发条件：重试 N 次后仍失败
  - 处理：人工介入 / 定时扫描 / 自动补偿
- 监控指标：
  - 生产速率 / 消费速率 / 堆积量
  - 消费延迟（lag）
  - 消费失败率 / 死信量
  - 重平衡频率（消费者上下线）
- 告警：
  - 堆积量 > 阈值
  - 消费延迟 > 阈值
  - 消费失败率 > 阈值

---

## 7. 验收清单（DoD）

- [ ] 生产者发送失败有补偿/告警，不静默丢失
- [ ] 消费者幂等方案明确（去重 key + 存储方式）
- [ ] 异常消息进死信队列，有人工介入入口
- [ ] 消费速率/并发有控制，不会打爆下游
- [ ] 监控告警覆盖：堆积、延迟、失败率、重平衡
