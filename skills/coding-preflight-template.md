# 编码前检查模板（Preflight）

> 目标：在写代码前把关键约束一次问清，避免返工与漏项。每次开始一个需求/任务，复制本模板填写。

## 门禁（强制）

- 开写前必须已有 PRD（至少包含：范围、功能点、验收标准、失败场景）
- 若 PRD 缺失信息：必须先补到 PRD 的 Open Questions，不允许直接猜测实现

---

- 任务名称：
- 业务目标（1 句话）：
- 非目标（明确不做什么）：
- 影响范围（哪些模块/服务/页面）：
- 预计上线方式：灰度 / 全量 / 仅内部

---

## 2. 输入 / 输出定义

### 输入（Inputs）
- 输入来源：HTTP API / 消息队列 / 定时任务 / 用户交互 / 文件上传
- 输入格式：JSON / Form / Protobuf / CSV / 其他
- 输入字段与约束：
  - 必填字段：
  - 字段类型：
  - 长度/范围：
  - 默认值：
  - 允许为空：

### 输出（Outputs）
- 输出对象：响应 / 事件 / 数据库变更 / 文件生成
- 输出格式：
- 输出字段与约束：
- 错误输出规范（error code / message / detail / request_id）：

---

## 3. 约束与边界条件（必须写）

- 权限/鉴权：
  - 需要登录？
  - 角色/权限模型？
- 幂等性：
  - 是否可能重复请求/重复事件？幂等 key 是什么？
- 一致性：
  - 强一致 / 最终一致？
  - 是否需要事务？
- 并发与竞态：
  - 是否可能并发更新同一资源？
  - 锁策略（乐观锁/悲观锁/分布式锁）？
- 超时与重试：
  - 外部依赖调用超时（ms）：
  - 重试次数与退避策略：
- 性能目标：
  - P95 延迟：
  - QPS/吞吐：
  - 数据规模：
- 兼容性：
  - API 版本：
  - 前后端兼容策略：
- 合规/安全：
  - 敏感字段（PII/密钥）：
  - 日志是否需要脱敏：

- [ ] 并发更新冲突

- [ ] 数据库写入失败/唯一约束冲突
