# 运维与 SRE 能力模板（sre-devops）

> 目标：部署稳定、发布可控、故障可演练、系统可观测，覆盖 CI/CD、容器编排、发布策略、可观测性、故障演练与混沌工程。

---

## 固定输出物（AI 必须产出）

- 基础设施即代码清单（IaC：Terraform/Pulumi/CloudFormation/Ansible）
- CI/CD 流水线设计（构建/测试/安全扫描/部署/回滚/审批）
- 发布策略（蓝绿/金丝雀/滚动/特性开关/灰度/AB 测试发布）
- 可观测性方案（Metrics/Logs/Traces/Alerts/SLO/SLI/Error Budget）
- 故障演练与混沌工程计划（场景/范围/回滚/观测/复盘）
- 容量规划与弹性伸缩策略（基线/峰值/预测/自动扩缩容）
- 应急响应手册（Incident 角色分工/沟通渠道/状态通告模板/联系人清单/incident 判定标准）
- 事件指挥机制（Incident Commander / Comms Lead / Ops Lead）与交接规则

## 必须/禁止规则

- 必须：所有环境（开发/测试/预发/生产）配置代码化，禁止手工改生产环境
- 必须：发布必须有回滚策略，且回滚时间 < 恢复时间目标（RTO）
- 必须：生产变更必须经过预发验证 + 自动化测试 + 人工审批（关键变更）
- 必须：可观测性数据必须保留 ≥ 可恢复时间窗口（建议 7-30 天）
- 必须：故障演练必须有明确的终止条件（自动/手动）和回滚预案
- 禁止：直接在生产环境手工执行未经验证的命令或配置变更
- 禁止：无 SLO/SLI 就承诺可用性（必须量化）
- 禁止：发布无灰度/无监控/无回滚方案直接全量上线
- 禁止：忽略告警或随意关闭告警（必须处理 root cause）

## 1. 基础设施即代码（IaC）

- 工具选型：
  - 云资源：Terraform / OpenTofu / Pulumi / CloudFormation
  - 配置管理：Ansible / SaltStack / Chef
  - 容器编排：Kubernetes（Helm / Kustomize / Operator）
  - 镜像构建：Dockerfile / BuildKit / Kaniko / Buildpacks
- 版本管理：
  - IaC 代码与业务代码同仓库或独立仓库，必须有 PR/MR 审批
  - Terraform state 远程存储（S3/Consul/Terraform Cloud），加锁防并发
- 环境一致性：
  - 开发/测试/预发/生产使用同一套 IaC 模板，仅变量差异
  - 环境漂移检测（定期对比实际 vs 代码定义）

## 2. CI/CD 流水线

### 2.1 阶段设计

```
代码提交 → 依赖安装 → Lint/Format → 单元测试 → 构建 → 安全扫描 → 集成测试 → 镜像构建 → 推送仓库 → 部署预发 → 验收测试 → 人工审批 → 生产发布 → 监控验证
```

- 并行：Lint + 单元测试 可并行
- 缓存：依赖缓存、构建缓存、Docker layer cache
- 产物：每个阶段产物可复用（jar/wheel/npm tar/docker image）

### 2.2 安全扫描

- 依赖漏洞：Snyk / Trivy / OWASP Dependency-Check
- 镜像扫描：Trivy / Clair / Snyk Container
- 密钥扫描：GitLeaks / TruffleHog（预提交钩子 + CI 双保险）
- SAST：SonarQube / Semgrep / CodeQL

### 2.3 审批与合规

- 自动发布：开发/测试环境
- 人工审批：预发 → 生产（关键服务）
- 发布窗口：明确禁止发布时间（如高峰期/节假日）

## 3. 发布策略

| 策略 | 适用场景 | 风险 | 回滚时间 |
|------|---------|------|---------|
| 滚动更新 | 无状态服务、兼容变更 | 中 | 分钟级 |
| 蓝绿部署 | 关键服务、数据库 schema 不变 | 低（快速切换） | 秒级 |
| 金丝雀/灰度 | 新功能验证、A/B 测试 | 低（可控范围） | 分钟级 |
| 特性开关 | 功能渐进开放、紧急关闭 | 低 | 秒级（开关切换） |

- 发布检查清单：
  - [ ] 变更内容已 review
  - [ ] 预发环境验证通过
  - [ ] 监控/告警已配置且正常
  - [ ] 回滚脚本/镜像已准备
  - [ ] on-call 人员已通知

## 4. 可观测性（Observability）

### 4.1 三大支柱

- **Metrics**：Prometheus / Grafana / Datadog / CloudWatch
  - RED 方法：Rate（请求率）、Errors（错误率）、Duration（延迟）
  - USE 方法：Utilization（利用率）、Saturation（饱和度）、Errors（错误）
- **Logs**：结构化日志（JSON）+ 集中采集（ELK / Loki / Splunk）
  - 必须字段：timestamp / level / service / trace_id / request_id / message
- **Traces**：OpenTelemetry + Jaeger / Zipkin / Tempo
  - 采样策略：尾部采样（Tail-based）保留异常链路，头部采样（Head-based）控制成本

### 4.2 SLO/SLI/Error Budget

- SLI（指标）：如 "p99 延迟 < 200ms"、"可用性 ≥ 99.9%"
- SLO（目标）：如 "季度可用性 99.95%"
- Error Budget：1 - SLO，如 0.05%  downtime budget
- 告警原则：
  - 基于 SLO  burn rate（消耗速度）告警，而非固定阈值
  - 分级：P0（立即处理）/ P1（30min 内）/ P2（4h 内）/ P3（次日）

## 5. 容量规划与弹性伸缩

- 基线容量：日常 QPS/TPS/CPU/内存/带宽的 p95
- 峰值预测：基于历史数据 + 业务事件（促销/节假日）
- 伸缩策略：
  - HPA（水平 Pod 自动伸缩）：CPU/内存/自定义指标
  - VPA（垂直 Pod 自动伸缩）：资源调整（注意重启）
  - Cluster Autoscaler：节点级扩缩容
  - 定时伸缩：预知高峰（如双 11）
- 容量测试：压测到目标容量的 1.5-2 倍，确认瓶颈

## 6. 故障演练与混沌工程

### 6.1 演练场景

- 实例级：随机杀 Pod/VM（Chaos Monkey）
- 网络级：延迟/丢包/分区（Chaos Mesh / ToxiProxy / Gremlin）
- 依赖级：数据库/缓存/消息队列故障注入
- 负载级：流量突增/慢请求/死锁

### 6.2 演练流程

1. 定义假设："如果 X 故障，系统应在 Y 时间内降级/恢复"
2. 确定范围：仅限预发/可控生产子集（有熔断/开关）
3. 执行注入：自动化或手动触发
4. 观测验证：监控是否触发预期告警/自动恢复
5. 终止与回滚：达到目标或异常扩大时立即终止
6. 复盘：更新 runbook / 修复发现的问题

### 6.3 禁止

- 无回滚预案就注入生产环境故障
- 在高峰期或无 on-call 覆盖时演练
- 注入超出预期范围的故障（如全集群网络分区）

## 7. 应急响应与 on-call

### 7.0 事件指挥与沟通（Incident Management）

- 角色定义（发生 incident 必须明确角色，避免“所有人都在做同一件事”）：
  - Incident Commander（指挥）：组织协同、分配任务、推动缓解优先
  - Comms Lead（沟通）：对内/对外状态更新、联系人通知、信息汇总
  - Ops Lead（操作）：执行缓解动作、回滚、变更操作与验证
- 委派与升级：
  - on-call 必须知道“可以委派”和“可以升级”，禁止一个人扛所有工作
  - 优先缓解（mitigation-first），根因分析可以在恢复后继续

### 7.1 预先准备（必须在事故前完成）

- 沟通渠道：预先约定并演练（Slack/电话会议/工单），禁止事故中临时决定
- 联系人清单：预先维护（业务owner/安全/PR/法务/供应商/云厂商），确保可一键通知
- 事件判定标准：
  - 明确什么情况升级为 incident（如：核心 SLO burn rate、核心功能不可用、数据一致性风险、支付链路异常）
  - 事件分级与响应时间与升级路径要与 on-call 制度一致
- 状态通告模板：预制 2-3 个模板，避免高压下临时写公告
  - 对内模板：现象/影响范围/当前缓解动作/下一次更新时间/负责人
  - 对外模板：确认事件/影响范围/临时措施/预计恢复时间（如可给）/后续更新渠道

### 7.1 分级响应

| 级别 | 场景 | 响应时间 | 升级路径 |
|------|------|---------|---------|
| P0 | 核心服务不可用/数据丢失 | 5 min | 立即电话 + 群 |
| P1 | 功能严重降级/大量用户影响 | 15 min | 电话/消息 |
| P2 | 局部功能异常/性能下降 | 1 h | 消息 |
| P3 | 非紧急问题/优化项 | 1 day | 工单 |

### 7.2 事后复盘（Postmortem）

- 模板：
  - 时间线（精确到分钟）
  - 影响范围（用户/交易/数据）
  - 根因分析（5 Whys）
  - 修复措施（短期/长期）
  - 预防措施（监控/告警/自动化）
- 原则： blameless（不追责人，只追责流程与系统）

## 8. 验收清单（DoD）

- [ ] IaC 已覆盖所有环境，state 远程存储加锁
- [ ] CI/CD 流水线包含：构建/测试/安全扫描/部署/回滚
- [ ] 发布策略明确（蓝绿/金丝雀/滚动），有回滚方案且回滚时间 < RTO
- [ ] 可观测性：Metrics（RED/USE）+ Logs（结构化）+ Traces（采样策略）+ 告警分级
- [ ] SLO/SLI 已定义，Error Budget 可追踪
- [ ] 容量规划有基线/峰值/伸缩策略，且通过压测验证
- [ ] 故障演练计划已执行 ≥1 次，有复盘记录和修复项
- [ ] on-call 分级/升级路径/事后复盘模板已就绪
- [ ] 事件指挥机制已落地（Incident Commander/Comms Lead/Ops Lead），并完成至少一次演练
- [ ] 沟通渠道、联系人清单、事件判定标准、状态通告模板已准备并可用
- [ ] 生产环境禁止手工变更，所有变更走流水线 + 审批
