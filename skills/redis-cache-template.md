# Redis 与缓存模板（redis-cache）

> 目标：缓存设计、一致性、高可用一次做对，避免穿透/击穿/雪崩和脏数据。

---

## 固定输出物（AI 必须产出）

- 缓存分层策略（本地/分布式/多级）+ 数据流图
- 缓存 Key 规范（前缀/版本/业务标识）+ 过期策略
- 缓存一致性方案（Cache-Aside / Read-Through / Write-Through / 延时双删）
- 防穿透/击穿/雪崩策略（布隆过滤器/互斥锁/随机过期/熔断）
- Redis 部署架构（单实例/主从/哨兵/Cluster）+ 容量评估

## 必须/禁止规则

- 必须：所有缓存 key 有明确过期时间（禁止永不过期）
- 必须：写操作先写 DB 再删/更新缓存（或明确延时双删）
- 必须：热点 key 有降级/本地缓存兜底
- 禁止：缓存当数据库用（大 value、复杂结构、持久化依赖）
- 禁止：Redis 事务包远程调用（管道内只做 Redis 操作）
- 禁止：keys * / flushall 在生产环境使用

---

## 1. 缓存分层设计

- 本地缓存：Caffeine / Guava Cache（进程内，无网络开销）
  - 适用：配置、字典、热点只读数据
  - 限制：容量、过期、无分布式一致性
- 分布式缓存：Redis（跨进程/跨服务）
  - 适用：会话、热点数据、计算结果、限流计数
- 多级缓存：本地 + Redis + DB
  - 数据流：本地 miss → Redis → DB → 回填 Redis + 本地
  - 一致性：本地 TTL < Redis TTL，主动失效/监听

---

## 2. Key 设计与规范

- 命名：`业务域:模块:标识:版本`（如 `user:profile:12345:v1`）
- 禁止：无命名空间的大 key（> 1MB）
- 过期策略：
  - 基础数据：TTL = 业务低频变更周期（如 5min）
  - 热点数据：TTL + 随机偏移（防雪崩）
  - 会话/token：严格过期 + 续期策略

---

## 3. 缓存一致性

- Cache-Aside（最常用）：
  - 读：先缓存 → 无则 DB → 回填缓存
  - 写：先 DB → 再删缓存（或更新缓存）
  - 脏读窗口：DB 写完成到缓存删除之间
- 延时双删（高一致）：
  - 删缓存 → 写 DB → 延时（如 500ms）→ 再删缓存
-  Canal / MQ 异步删：
  - DB binlog → MQ → 消费删缓存（最终一致）
- 禁止：先删缓存再写 DB（并发读易脏）

---

## 4. 防穿透 / 击穿 / 雪崩

- 缓存穿透（查询不存在数据）：
  - 布隆过滤器（预加载合法 key）
  - 空值缓存（TTL 较短，如 30s）
- 缓存击穿（热点 key 过期瞬间）：
  - 互斥锁（Redis SETNX / 本地锁，只允许一个线程回源）
  - 逻辑过期（不删 key，后台异步刷新）
- 缓存雪崩（大量 key 同时过期）：
  - 随机 TTL（基础 TTL + random(0, 30%)）
  - 熔断降级（DB 压力大时返回默认值）
  - 多级缓存分担

---

## 5. Redis 部署与高可用

- 架构选择：
  - 单实例：开发/测试
  - 主从 + 哨兵：中小型，自动 failover
  - Cluster：大规模，数据分片（16384 slots）
- 容量评估：
  - 数据量 = key 数 × 平均 value 大小 × 副本数
  - 内存预留：maxmemory + 淘汰策略（allkeys-lru / volatile-lru）
- 持久化：
  - RDB：定时快照，恢复速度快
  - AOF：实时日志，数据更完整（everysec）
  - 混合：RDB + AOF（Redis 4.0+）
- 禁止：无持久化直接用于重要数据（或明确可重建）

---

## 6. 性能与监控

- 慢查询：slowlog，> 10ms 告警
- 大 key 扫描：redis-cli --bigkeys / rdb-tools
- 连接池：maxactive/maxidle/timeout/等待队列
- 监控指标：
  - 内存使用率、碎片率
  - QPS、命中率
  - 主从复制延迟
  - 阻塞客户端数

---

## 7. 验收清单（DoD）

- [ ] 所有缓存 key 有 TTL，热点 key 有随机偏移
- [ ] 写操作一致性策略明确（Cache-Aside + 延时双删 / MQ 异步）
- [ ] 穿透/击穿/雪崩至少各有一种防护手段
- [ ] 大 key 扫描无异常，内存容量有预留
- [ ] 部署架构匹配业务规模，有 failover 能力
