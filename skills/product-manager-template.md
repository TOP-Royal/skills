# 产品经理能力模板（product-manager）

> 目标：从开发视角补全产品思维，把"接需求"升级为"参与需求定义"——能反向推导用户价值、评估可行性、推动产品决策。

---

## 固定输出物（AI 必须产出）

- 用户故事清单（作为…我想要…以便…）+ 优先级（MoSCoW）
- 竞品对标表（功能/体验/定价/技术架构对比）
- 需求价值评估（用户量/频次/痛点强度/开发成本/预期收益）
- 产品路线图（MVP/近期/中期/远期，版本节点）
- 数据指标定义（北极星指标 + 过程指标 + 埋点清单）
- 风险与可行性评估（技术风险/合规风险/资源风险/市场风险）

## 必须/禁止规则

- 必须：每个功能都有"用户价值假设"（不是老板说要做就做）
- 必须：需求有明确的"成功指标"（可量化，不是"提升体验"）
- 必须：MVP 范围可裁剪，优先验证核心价值（不要一次性做大而全）
- 禁止：没有竞品调研就定义全新交互/全新品类（除非明确创新点）
- 禁止：需求文档只有功能列表，没有用户场景和失败分支
- 禁止：功能上线后没有定义"何时下线/迭代"的指标阈值

### 2.1 需求清单与优先级

|------|--------|---------|---------|---------|---------|--------|
| | | | | | | |

- 优先级规则：
  - **Must Have**：不做用户无法完成核心任务
  - **Should Have**：显著提升体验或效率
  - **Could Have**：有更好，没有也能用
  - **Won't Have**：明确本次不做

### 2.2 价值公式（简化版）

- 价值 = （用户数 × 频次 × 痛点强度） / （开发成本 + 维护成本 + 机会成本）
- 先做"高价值、低复杂度"，再考虑"高价值、高复杂度"
