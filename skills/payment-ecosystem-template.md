# 支付生态能力模板（payment-ecosystem）

> 目标：在真实业务中把支付接对、接稳、可对账、可追责、可风控、可合规。覆盖国内（微信支付/支付宝/银联/云闪付/银行）与海外（Stripe/PayPal/Apple Pay/Google Pay）常见落地。

---

## 固定输出物（AI 必须产出）

- 支付域模型（Order/Payment/Refund/Settlement/Dispute/Account）与状态机
- 通道选型与路由策略（微信/支付宝/银联/Stripe/PayPal…，按地区/币种/费率/成功率/风控）
- 交易一致性方案（幂等键、重试、补偿、最终一致、对账驱动纠错）
- 回调（Webhook/Notify）处理规范（验签、去重、顺序、延迟、降级）
- 风控与合规清单（反洗钱/反欺诈/隐私/PCI/商户资质/条款）
- 对账与结算方案（账单来源、差异处理、日切、资金流水、会计科目映射）
- 观测与告警（成功率、超时率、回调延迟、差错率、通道可用性）
- 测试矩阵（沙箱/灰度/真机/边界/异常/压测）+ 回归用例

## 必须/禁止规则

- 必须：**所有支付相关接口幂等**（下单/支付/退款/取消/查询/回调处理）
- 必须：**回调必须验签 + 去重 + 可重放**，并记录 `request_id`/`payment_id`/`provider_txn_id`
- 必须：所有第三方调用设置**超时**，且有**重试上限**与退避策略（避免雪崩）
- 必须：所有金额用**整数最小货币单位**（如分/cents），禁止浮点
- 必须：对账是主线：以**对账结果驱动纠错**，避免只靠“实时回调”自嗨
- 必须：敏感信息（卡号/证件/密钥）不落日志、不落库明文；密钥分级、轮换、审计
- 禁止：在服务端信任前端传的支付状态/金额/币种
- 禁止：回调里做重逻辑或长事务（必须快速 ack，异步处理）
- 禁止：直接把第三方错误原样返回给客户端（要做错误映射与脱敏）

## 2. 支付域模型与状态机

### 2.1 核心对象

- `Order`：业务订单（买了什么）
- `Payment`：支付单（怎么付、付没付）
- `Refund`：退款单（退多少、退到哪）
- `Settlement`：结算（平台/商户的资金结算）
- `Dispute/Chargeback`：拒付/争议（海外重点）
- `Account/Ledger`：内部记账账户与流水（可选但强烈建议）

### 2.2 Payment 状态机（示例）

- CREATED → PENDING → SUCCEEDED
- CREATED/PENDING → FAILED
- PENDING → CANCELED（超时取消）
- SUCCEEDED → REFUNDING → REFUNDED（部分/全额）

**要求**：
- 所有状态变更必须有事件源（回调/主动查询/对账纠错）
- 状态变更必须幂等（同一事件重复投递不会产生副作用）

---

## 3. 下单/支付链路设计（强制幂等）

### 3.1 幂等键设计

- 下单：`idempotency_key = user_id + cart_hash + timestamp_bucket`
- 发起支付：`idempotency_key = payment_id`（推荐）
- 退款：`idempotency_key = refund_id`

### 3.2 第三方调用超时与重试

- 超时：连接/读写分离（如 1s/3s/3s）
- 重试：仅对**可重试**错误（网络超时/网关 5xx）重试
- 退避：指数退避 + 抖动
- 上限：总重试次数与总耗时上限（避免拖垮线程池）

### 3.3 客户端返回策略

- 发起支付只返回：`payment_id` + `provider_payload`（SDK/收银台参数）
- 不返回第三方敏感字段；错误码做映射（见第 8 节）

---

## 4. 回调（Webhook/Notify）处理（强制）

### 4.1 安全校验

- 验签：微信/支付宝/Stripe/PayPal 的签名校验
- 防重放：校验时间戳/nonce（通道支持时）
- 白名单：IP 白名单（可选，注意云环境 IP 变化）

### 4.2 去重与顺序

- 去重：以 `provider_event_id`/`provider_txn_id` 做唯一键
- 顺序：同一 `payment_id` 的事件串行处理（队列/锁/单线程分片）

### 4.3 快速 ACK + 异步处理

- 回调入口：只做验签、落库（原始报文）、投递事件队列、返回成功
- 异步 Worker：执行状态机推进、发放权益、写账、通知业务

---

## 5. 主动查询与“以对账驱动纠错”

### 5.1 主动查询（Query）

- 场景：
  - 客户端支付完成但回调未达
  - 订单超时但第三方已扣款
  - 退款长时间未完成
- 策略：
  - 支付发起后：T+30s/T+2m/T+10m 轮询（上限）
  - 退款：更长轮询窗口

### 5.2 对账（Reconciliation）

- 输入：通道日账单（微信/支付宝/银联/Stripe 报表）+ 内部流水
- 输出：差异单（少记/多记/金额不一致/状态不一致）
- 处理：自动纠错 + 人工工单
- 日切：明确“账务日”与时区（北京时间/UTC），防跨日差异

---

## 6. 退款、部分退款与售后

- 退款规则：
  - 可退金额 ≤ 已支付金额 - 已退款金额
  - 部分退款：多笔 `Refund` 关联同一 `Payment`
  - 原路退：通道限制与退款路径说明
- 幂等：同一 `refund_id` 多次调用只能产生一次外部退款
- 退款状态：CREATED → PROCESSING → SUCCEEDED/FAILED
- 退款失败兜底：人工处理 + 二次发起（需风控）

---

## 7. 风控与合规（真实必备）

- 反欺诈：
  - 设备指纹/地理位置/异常 IP
  - 频控：同卡/同设备/同用户的支付频次限制
  - 黑名单：用户/设备/卡号散列
  - 风险评分：阈值拦截/二次验证（短信/3DS/人审）
- 合规：
  - PCI DSS：如你触碰卡号/持卡人数据（尽量不要，交给 PSP）
  - 隐私：最小化采集、脱敏、加密、授权
  - 资金合规：商户资质、对公账户、分账合规（如涉及）
  - 条款：退款政策、争议处理、订阅取消

---

## 8. 错误码与用户提示（稳定映射）

- 内部错误码：`PAY_xxx`
- 必须输出：`code` + `message` + `request_id`
- 映射原则：
  - 通道错误（验签失败/余额不足/风控拦截/超时）→ 统一错误域
  - 用户可操作 vs 不可操作：给不同提示与引导（重试/换卡/联系客服）

---

## 9. 观测与告警（上线必备）

- 指标：
  - 支付成功率（按通道/地区/产品）
  - 超时率/重试率
  - 回调延迟（P50/P95/P99）
  - 对账差错率
  - 退款成功率与时延
- 日志：全链路 `request_id`，关键字段脱敏
- 告警：
  - 成功率突降、回调积压、对账差错突增、通道不可用

---

## 10. 测试矩阵（必须真实）

- 沙箱：微信/支付宝沙箱（如可用）、Stripe test mode、PayPal sandbox
- 真实联调：
  - 小额真实支付（可退款）
  - 异常场景：超时/断网/重复回调/回调乱序
- 账务测试：
  - 对账文件解析正确
  - 差异单可重放可修复
- 压测：
  - 回调峰值吞吐
  - 下单/支付 API 的 p95 延迟

- [ ] 回调验签 + 去重 + 可重放；回调入口快速 ACK，异步处理落地
- [ ] 所有第三方调用有超时 + 重试上限 + 退避策略
- [ ] 金额使用整数最小货币单位，无浮点
